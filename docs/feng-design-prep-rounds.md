# feng 设计前复习草稿轮次

本文记录进入 feng 系统概要设计前的 5 轮预备循环。每轮都必须包含：

```text
复习学习结果 -> 看草稿 -> 反思 -> 写新草稿
```

这些草稿不是最终概要设计。它们用于防止 feng 被调研仓库牵着走，也用于把产品概念从“想法”收束到后续可设计的判断。

## 轮次状态

| 轮次 | 状态 | 产物 |
| --- | --- | --- |
| 第 1 轮 | 已完成 | 草稿 v1：feng 不是 agent creator，而是智能行为成长系统 |
| 第 2 轮 | 已完成 | 草稿 v2：file-native 是事实层，message list 是编译产物 |
| 第 3 轮 | 已完成 | 草稿 v3：hatch 是能力包，不是成长历史复制 |
| 第 4 轮 | 已完成 | 草稿 v4：反馈回流是归因和采纳流程，不是自动上游吸收 |
| 第 5 轮 | 已完成 | 草稿 v5：概要设计应围绕成长闭环展开，而不是功能全集 |

## 第 1 轮

### 复习学习结果

调研总结的核心判断是：优秀 agent 不是靠更多功能取胜，而是把模型行动放进一个可观察、可约束、可验证、可复制的运行世界。聊天历史不是能力，任务推进不是能力成熟，记录经验也不是自我演进。

对 feng 最重要的学习是：grow 必须把模糊目标变成文件化证据、运行契约、目标世界接口和可验证候选；hatch 必须把稳定能力从原始成长历史里分离出来。

### 看草稿

当前 `docs/product-concept.md` 已经明确：

```text
feng 是 file native 的智能行为成长系统。
feng 没有用户需要理解的 session 概念，一个 grow 单元是连续成长空间。
grow 的对象不是固定 LLM loop。
hatch 产物可以是命令、服务、游戏组件、行为树、状态机、LLM agent 或混合运行单元。
反馈回流必须归因，不能无脑向上游吸收。
默认反馈路由 skill 是通用能力，但基础协议要稳定、场景策略可 grow。
```

### 反思

概念文档当前最大风险不是方向错，而是“描述已经很完整，容易误以为设计也完成了”。实际上它仍然在产品概念层：没有定义 grow 单元内部状态、message list 编译、反馈单元、hatch 包边界和验证机制。这是合理的，因为现在还没进入设计。

另一个风险是“智能行为成长系统”听起来仍可能被误读成 agent creator。后续概要设计必须持续强调：feng 不是选择模板生成 agent，而是在目标世界中塑造可运行行为。

### 新草稿 v1

```text
feng 的系统设计不应从“要有哪些 agent 类型”开始，而应从“一个智能行为如何被塑造成可运行能力”开始。

设计对象不是 agent 模板，而是 grow 单元。grow 单元保存目标、材料、目标世界接口、候选行为、验证证据、反馈归因、下一轮 message list 和 hatch readiness。

系统概要设计要回答：一个 grow 单元如何接收事实、编译下一轮模型输入、运行候选、收集证据、判断是否继续、以及如何把成熟能力 hatch 成目标世界可接入的包。

所有具体机制都必须服务这个闭环。工具、skills、任务、MCP、后台、cron、worktree、多 agent 都只能作为可选 harness 能力，不能成为 feng 的产品心智。
```

## 第 2 轮

### 复习学习结果

opencode、CodeWhale 和 learn-claude-code 共同证明：完整运行事实和模型可见 message list 必须分层。事件、transcript、工具输出、压缩摘要、memory、context source、provider request 是不同层，不应该混成一坨上下文。

Shinsekai 和 AssistantAgent 进一步说明：目标世界资产、reference、tool artifact、运行 trace、输出 contract 也不应该都变成 prompt 文本。它们可以影响下一轮模型输入，但必须先作为可检查事实存在。

### 看草稿

产品概念已经写了 file native 原则，并特别强调下一轮 LLM loop 的 message list 也是文件。这是一个强判断：message list 不是临时请求体，而是一等产物。

草稿仍没有明确区分：

```text
事实层：目标、材料、事件、证据、工具结果、反馈、候选包。
编译层：下一轮 message list、system prompt section、可见 skill、可见工具面。
执行层：本轮模型调用、工具执行、候选运行、验证。
```

### 反思

如果概要设计不先区分这三层，feng 会很容易滑向两种错误：

```text
把所有文件都塞进上下文，导致 file-native 变成上下文污染。
把 message list 当临时运行细节，导致用户无法检查 feng 下一轮到底基于什么 grow。
```

因此后续设计必须把 message list compiler 放到核心位置，但不能把它设计成“大 prompt 模板”。它应该从 grow 单元的事实层编译出当前轮可见表示，并留下编译依据。

### 新草稿 v2

```text
feng 的文件系统应至少在概念上分成事实层、编译层、执行层。

事实层保存完整成长依据：目标、材料引用、事件、反馈、工具结果、验证证据、候选能力、hatch readiness、版本和归因。

编译层保存当前轮模型可见表示：下一轮 message list、system prompt section、可见 skill 列表、可见工具面、被排除或压缩的依据。

执行层保存本轮动作结果：模型响应、工具调用、候选运行、错误恢复、验证报告、退出原因。

message list 是编译结果，不是真相本身。它必须可定位、可复查、可回放，但不能替代事实层。
```

## 第 3 轮

### 复习学习结果

Shinsekai 的导入导出、Tauri 资源和 runtime_manifest 说明：可复制需要资源、依赖、版本、诊断和测试门槛。AssistantAgent 的 skill package/experience 管理说明：包要能预览、导入、导出、处理冲突、保留 references/assets/metadata。opencode 的 schema changelog 说明：对外契约会演进，需要版本和兼容边界。

这些经验都指向同一件事：可复制不是复制当前工作目录，而是提取稳定能力，并声明它的运行条件。

### 看草稿

产品概念的 hatch 段落已经明确：hatch 不是打包聊天历史，也不一定导出 LLM agent。它强调输入、输出、运行环境、可感知内容、可调用动作、调试、验证和后续 grow。

草稿还可以更强地表达一个约束：hatch 包必须排除原始成长噪声、失败尝试、本机秘密、无关材料和未采纳候选。否则“可复制”会带来污染和泄密。

### 反思

如果后续概要设计把 hatch 直接做成“把 .feng 复制到目标目录”，它会违背调研结果。grow 目录包含成长过程，hatch 包包含稳定能力，两者是不同东西。

同时，不能过早规定 hatch 包一定是某种目录布局或 manifest schema。现在只应该写清楚概念边界和必须携带的信息，具体文件格式留到设计文档后段或后续阶段。

### 新草稿 v3

```text
hatch 的系统含义是从 grow 单元中提取稳定能力，形成目标世界可接入的能力包。

hatch 包应该包含：运行入口、输入契约、输出/事件契约、所需资源、权限边界、可观察日志、反馈路由、验证报告、版本和来源。

hatch 包不应该包含：未采纳候选、失败草稿、本机密钥、原始长聊天、无关材料、临时上下文、不可解释的缓存。

grow 目录是成长事实库，hatch 包是可运行交付物。二者可以互相引用，但不能混为一体。
```

## 第 4 轮

### 复习学习结果

Hermes 的 observer/action 分离很关键：观察到问题不等于能修改事实，阻断、改写、审批、采纳是不同权力。AssistantAgent 的 Experience 也提醒：经验条目只是能力候选，必须有披露、置信度、来源、工具关联和采纳边界。

learn-claude-code 的 inbox、task_notification、cron queue 都是外部事件入口，但它们只说明“消息可以进来”，没有解决“哪些消息应该进入 grow”的问题。

### 看草稿

产品概念已经写了可观察性、反馈路由、默认反馈路由 skill 和多层演进闭环。它明确“运行问题不能无脑上报，也不能无脑吸收”。

草稿还需要在后续概要设计中把反馈路由拆成更清晰的阶段，而不是只写成一段理念：

```text
收集。
归因。
形成候选。
附证据。
决定本地处理、上游提议、忽略或等待人工确认。
验证。
版本化发布。
```

### 反思

“多层循环”很容易被误解成自动递归升级：libai 的问题自动进 xiaoshuo，xiaoshuo 的问题自动进 feng。这会污染上游，也会让 feng 越 grow 越偏。

更合理的概念是：下游只能产生反馈候选，上游是否吸收必须由本层的反馈路由 skill、证据、版本影响和验证门槛决定。

默认反馈路由 skill 的基础协议应该稳定，场景策略可以 grow。否则下游 agent 可以修改自己的上报规则来隐藏失败，或者把噪声伪装成系统性问题。

### 新草稿 v4

```text
反馈回流不是自动学习，而是一个受控采纳流程。

每个 hatch 产物都应能产生反馈单元。反馈单元至少表达：来源、版本、运行场景、观察到的问题、证据、初步归因、建议流向和隐私边界。

反馈单元进入 grow 后，默认只能成为候选。候选可以被本地吸收、上游提议、拒绝、忽略、等待更多证据或请求人工确认。

默认反馈路由 skill 应包含稳定基础协议和可 grow 的场景策略。基础协议负责可追踪和边界，场景策略负责判断哪些信号有意义。

上游吸收必须经过证据和验证，不允许下游问题无条件污染上游 feng。
```

## 第 5 轮

### 复习学习结果

全部调研的共同压力是：优秀项目都很容易诱导 feng 走向“功能全集”。CodeWhale 诱导成 coding runtime，opencode 诱导成 session/state API，Hermes 诱导成平台，Shinsekai 诱导成领域应用，AssistantAgent 诱导成企业框架，learn-claude-code 诱导成 Claude Code harness 复刻。

真正可迁移的不是它们的产品表面，而是状态、边界、证据、契约、分层和生命周期。

### 看草稿

前 4 轮草稿已经形成几个约束：

```text
系统设计从 grow 单元开始。
file-native 分事实层、编译层、执行层。
message list 是编译产物。
hatch 是稳定能力包。
反馈回流是候选采纳流程。
工具、skills、任务、MCP、后台、cron、worktree、多 agent 都不是产品第一心智。
```

产品概念文档也强调当前不是详细设计阶段，非目标中排除了目录 schema、CLI 完整设计、provider adapter、MCP adapter、eval runner 等低层细节。

### 反思

概要设计文档接下来必须小心：它应该比产品概念更工程化，但不能直接进入详细实现。合理范围是：

```text
系统的核心对象。
核心闭环。
关键子系统职责。
信息如何流动。
哪些东西必须文件化。
哪些边界必须稳定。
哪些能力是可选 harness，而不是核心产品心智。
第一阶段可行切面。
```

不应该在概要设计里确定：

```text
具体目录树。
具体 JSON/YAML schema。
完整 CLI 子命令。
具体 provider/MCP 实现。
完整 eval 框架。
插件市场或 GUI。
```

否则会提前把 feng 锁成一个框架，而不是继续服务“智能行为成长系统”的概念。

### 新草稿 v5

```text
feng 系统概要设计应该围绕一个主闭环展开：

目标进入 grow 单元。
grow 单元把目标、材料和反馈沉淀为事实层。
message compiler 从事实层编译下一轮模型可见输入。
grow runtime 执行候选生成、工具调用、候选运行和验证。
evidence/readiness 判断候选是否继续 grow 或进入 hatch。
hatch 从 grow 单元提取稳定能力包。
hatch 产物运行后产生反馈单元。
feedback router 将反馈归因为本地、上游、忽略或待确认。
被采纳的反馈进入下一轮 grow。

概要设计要说明这些对象和流程的职责边界，不急于定义具体文件格式。

系统复杂性应该被压在内部：用户仍然只感觉自己在提出一个智能行为、持续给材料和反馈，直到 feng 说“成了”。
```

## 进入概要设计前的最终边界

```text
可以开始写 feng 系统概要设计。
概要设计必须继承 file-native、单 grow 单元、无用户 session、message list 文件化、hatch 非固定 LLM loop、反馈候选采纳、多层回流 skill 这些概念。
概要设计不能把 feng 写成 agent creator、Claude Code 复刻、企业 agent 平台、插件/MCP 平台、任务调度器或多 agent 编排器。
```
