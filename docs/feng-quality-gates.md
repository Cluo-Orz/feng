# feng 质量门禁与不能漏题

本文补充 feng、xiaoshuo、libai-chongshengle 三层项目的质量门禁。它仍然是概念层说明，不定义最终文件 schema、评分器实现、CLI、eval runner 或具体目录结构。

## 核心判断

质量门禁不是由本文替 feng 写死的一套检查清单。真正的产品要求是：

```text
feng 必须在 grow 过程中自己产出质量门禁。
门禁必须来自用户目标、目标世界、材料、约束、反馈和 DoD。
门禁必须文件化、可追溯、可更新、可验证。
未覆盖的目标、材料或约束必须显式暴露，不能静默漏题。
没有门禁或门禁覆盖不完整时，不能 ready_to_hatch。
```

这就是“不能漏题”的含义。feng 不能只完成自己顺手想到的部分，也不能因为某个候选能力看起来不错就宣布成熟。它必须能说明：用户到底出了哪些题、材料里隐含了哪些要求、目标世界要求它交付什么、当前门禁覆盖了哪些、还有哪些没有覆盖。

## 门禁是 grow 的产物

每个 grow 单元都应该在早期产出一个质量门禁草案，并在 grow 中不断修订。

这个门禁不是固定模板，而是一个持续成长的评价契约。它至少回答：

```text
这个智能行为要满足哪些显式目标？
用户没有明说但材料和目标世界隐含了哪些要求？
哪些要求是必须通过的硬门槛？
哪些要求是可优化但不阻塞 hatch 的软指标？
哪些要求必须人工确认？
哪些要求可以由工具、测试、模拟、运行 trace 或 LLM evaluator 辅助判断？
每个门禁项需要什么证据？
每个门禁项当前状态是什么：未覆盖、待验证、已通过、失败、被豁免、需要人工判断？
```

因此，门禁不是“告诉 feng 你要这样写小说”。更准确地说，feng 要能从“我要做一个小说 agent”“我要写《李白重生了》”这类目标中，自己推导出应该检查什么，并把推导依据留在文件中。

## 目标覆盖表

为了避免漏题，feng 需要维护目标覆盖表。

目标覆盖表不是详细 schema，而是一种概念要求：所有进入 grow 的目标、材料、约束和反馈，都必须被映射到某个处理状态。

典型状态包括：

```text
已转成门禁项。
已转成材料假设。
已转成运行契约。
已转成反馈路由规则。
等待更多信息。
明确不采纳，并记录原因。
不属于本层，应上报或留在下游。
```

如果用户说“我要一个写小说 agent”，feng 不能只生成 prompt。它至少要覆盖：写什么、接收什么材料、输出什么、如何处理长篇上下文、如何接收作者反馈、如何判断章节质量、如何区分作品事实和通用写作能力、如何参与后续 grow。

如果用户说“写《李白重生了》”，xiaoshuo 不能只写一章。它至少要覆盖：题材承诺、主角设定、重生机制、时代/世界观、章节目标、人物关系、风格边界、已写事实、作者反馈、发布或继续创作标准。

## 门禁层级

feng 多层闭环至少有三类门禁。

### 系统层门禁

系统层门禁判断 feng 自己是否具备可信的成长能力。

它应该由 feng 在自身 grow 中产出和维护，示例方向包括：

```text
目标是否被完整解析成 grow 单元。
message list 是否文件化并可追溯。
每次 grow attempt 是否留下输入、动作、证据和退出原因。
readiness 是否基于证据而不是模型自信。
hatch 包是否排除了成长噪声、本地秘密和未采纳候选。
反馈路由是否能区分本地、领域、系统问题。
下游数据是否被隐私边界保护。
未覆盖要求是否会阻止 ready_to_hatch。
```

这些不是小说专属规则，而是 feng 的产品可信度门槛。

### 领域 agent 门禁

领域 agent 门禁判断 xiaoshuo 这类 hatch 产物是否能胜任一个领域。

它应该由 feng 在 `F:\code\xiaoshuo` 的 grow 中产出和维护。示例方向包括：

```text
是否定义了小说 agent 的目标世界接口。
是否能接收作品设定、提纲、章节、作者反馈和风格要求。
是否能输出章节、改稿、续写计划、设定冲突和修订建议。
是否能维护作品事实、当前章节上下文、长期写作策略和反馈候选的边界。
是否能把作者反馈转成可执行修订任务。
是否能在不同作品项目中复用，而不是只服务某一本小说。
是否能把失败归因到作品层、小说 agent 层或 feng 系统层。
是否有足够运行证据支持 hatch 成 xiaoshuo。
```

这些方向也不是固定评分表。真正的门禁项应由 xiaoshuo grow 时根据创作者目标、样例材料、目标作品类型和验证反馈具体生成。

### 作品层门禁

作品层门禁判断 libai-chongshengle 是否达到当前作品目标。

它应该由 xiaoshuo 在 `F:\code\libai-chongshengle` 中根据作品目标产出和维护。示例方向包括：

```text
题材承诺是否成立。
主角设定是否稳定。
重生机制、时代背景或世界规则是否自洽。
章节目标是否完成。
人物关系和动机是否前后一致。
已写伏笔、事件和设定是否被正确继承。
叙事节奏、场景、对白和文风是否符合作者目标。
作者反馈是否被处理或明确挂起。
当前章节是否可作为候选发布稿。
```

作品层有审美和作者意图。agent 可以给出结构性质量判断和候选发布建议，但不应该单独宣称“这就是好小说”。最终是否合格，至少在早期应该保留作者验收门槛。

## 不能漏题机制

“不能漏题”不是一句提示词，而是一组产品约束。

feng 或 hatch agent 在宣布通过前，必须能说明：

```text
输入目标有哪些。
材料中抽取了哪些显式要求。
材料中推导了哪些隐含要求。
哪些要求进入了质量门禁。
哪些要求没有进入门禁，原因是什么。
哪些门禁已通过，证据在哪里。
哪些门禁失败、待验证、被豁免或需要人工判断。
本次 ready/hatch/发布判断是否受到未覆盖项阻塞。
```

如果无法说明，结论只能是继续 grow、等待输入、请求确认或标记 blocked，不能宣布完成。

## 门禁与用户输入

用户可以补充门禁，但不应该被迫手工设计完整门禁。

好的体验应该是：

```text
用户提出目标。
feng 生成首版质量门禁。
用户可以删改、确认或补充。
feng 在运行和反馈中发现缺口，继续修订门禁。
每次 readiness 判断都引用当前门禁和证据。
```

这保留了简单产品心智：用户不是在配置 eval 框架，而是在塑造一个智能行为。复杂的质量判断由 feng 帮用户产出并维护。

## 门禁不能成为硬编码模板

小说场景可以有默认门禁 skill，但这个 skill 只能提供启发式框架，不能替代 grow 中的具体门禁生成。

风险在于：如果 feng 内置一张“小说评分表”，它可能会把所有小说都压成同一种写法，并漏掉当前作品真正重要的题。

因此：

```text
默认门禁 skill 提供维度，不直接给最终结论。
具体门禁必须绑定当前目标、材料、作者意图和目标世界。
门禁项要有来源。
门禁变更要有原因。
门禁豁免要有记录。
门禁失败要进入反馈或 grow 议程。
```

这也是防止 feng 变成拼装产品的关键：学习优秀 agent 的 eval 思路，但不把某个场景的静态评价表当成 feng 的本质。

## 三层案例中的门禁责任

在 `feng -> xiaoshuo -> libai-chongshengle` 中：

```text
feng 负责生成和验证系统层门禁。
feng 在 xiaoshuo grow 中推动 xiaoshuo 生成领域 agent 门禁。
xiaoshuo 在 libai-chongshengle 中生成作品层门禁。
libai-chongshengle 的作品层门禁结果可以反馈给 xiaoshuo。
只有归因为系统性问题的门禁缺陷，才提议回流给 feng。
```

比如：

```text
作品漏掉第三章伏笔：优先是 libai-chongshengle 作品层门禁失败。
多部作品都出现长篇事实遗忘：可能是 xiaoshuo 领域 agent 门禁失败。
trace 无法定位本轮 message list：是 feng 系统层门禁失败。
反馈路由把作品原文直接上报：是 feng 默认反馈 skill 或隐私门禁失败。
```

这样，质量判断和反馈回流才不会混层。

## 当前边界

本文明确质量门禁必须存在，但不在概念阶段规定：

```text
门禁文件格式。
评分算法。
LLM evaluator prompt。
样例集规模。
自动化测试 runner。
人工验收 UI。
发布评分阈值。
具体章节评审标准。
```

这些应该进入后续详细设计。概念阶段要稳定的是：门禁必须由 feng 产出，必须覆盖目标，必须留下证据，不能漏题，不能靠模型自信宣布合格。
