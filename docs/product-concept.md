# feng 产品概念设计文档

本文处于概念梳理阶段，不定义具体架构、文件 schema、工具协议、CLI 完整设计或 GUI。它的目标是把 feng 的产品终态、用户心智、成长闭环、可复制机制和可行性边界讲清楚，后续可以作为 README 的一部分。

## 一句话

feng 让一个模糊的智能行为，在真实世界中被塑造、成型、部署、观察、回流，并继续演进。

它不是普通 agent creator，不是聊天机器人生成器，也不是一套固定 agent 模板。feng 的核心不是“生成一个 agent”，而是让创作者不断提供目标、材料、反馈和运行结果，最终得到一个能嵌入目标世界、能被观察、能被迭代、能被复制的运行单元。

feng 也应该是一个 file native agent：运行中产出的关键内容都必须能在文件中找到。目标、材料、候选能力、运行记录、反馈、验证证据、hatch 产物，以及下一轮 LLM loop 将使用的 message list，都不应该只存在于一段不可追溯的进程内存或聊天窗口里。

file native 也包括成本和上下文健康的证据。feng 每次备份、重跑、清理或替换工作目录之前，都应该先从文件化 usage 和运行报告中分析缓存命中率。成熟的长程 coding agent 往往能让稳定前缀获得 80%-95% 甚至更高的缓存命中；如果 feng 在重复 grow/run 中只有 10% 左右甚至更低，不能把它当成正常成本，而要视为 Message Compiler、Prompt / Context Kernel 或 provider cache 观测存在系统性问题。

## 产品本质

feng 面向的是“智能行为”的成长。

用户最初可能只有一句话：

```text
我想做一个 boss agent。
我想做一个写小说 agent。
我想做一个会开小车的 agent。
我想让 feng 改进自己。
```

feng 的任务不是让用户先选择 agent 类型、填写模板、配置工具，而是在成长过程中逐步弄清楚：

```text
它要做什么。
它运行在哪个世界里。
它从哪里接收信息。
它如何处理信息。
它如何影响外界。
它怎样被观察。
它怎样证明自己可用。
它怎样把问题回流给自己或上游。
```

因此，feng 不是“形态优先”的创建器，而是“成长优先”的成型系统。agent 的最终形态应该从目标世界和运行反馈里长出来，而不是从一开始被模板锁死。

## File Native 原则

feng 的成长状态应该以文件为主要载体。文件不是实现细节，而是产品可信度的一部分：用户、feng 自己、下游 agent 和未来版本，都需要能回看一次成长为什么发生、用了哪些材料、产出了什么候选、为什么进入下一轮。

更准确地说，feng 应该避免“只有进程知道”的关键状态。运行中产出的内容、下一轮要使用的上下文、一次判断的证据、一次失败的原因，都应该能在工作区里找到对应文件。数据库、索引、缓存和内存状态可以存在，但不能成为关键成长状态的唯一真相来源。

因此，feng 运行中产生的关键内容应该可定位、可检查、可复用：

```text
用户输入和关键澄清。
当前目标和运行契约。
世界入口和材料引用。
候选能力和被排除的候选。
工具调用、运行结果和失败记录。
反馈归因和上报证据。
验证结果和 ready 判断。
hatch 产物及其版本边界。
下一轮 LLM loop 将使用的 message list。
```

这里的核心要求不是“把所有东西都永久保存”，而是不能让关键成长状态只停留在模型上下文、临时会话或不可检查的内存中。需要保留什么、保留多久、如何脱敏、如何压缩，是后续设计问题；但概念层必须成立：feng 的成长过程应该能通过文件被追踪和复现。

file native 也意味着下一轮 LLM loop 的输入不是凭空拼出来的。message list 应该是一个一等产物，而不是临时拼接出来后立刻消失的请求体。它需要能说明自己从哪些目标、材料、反馈、约束和历史证据编译而来。这样用户才能判断 feng 到底在基于什么继续 grow，也才能在出错时回放、审计或修正下一轮 grow 的依据。

缓存命中率是 message list 是否被稳定编译的外部信号。低命中率通常不是“模型调用贵一点”这么简单，而是说明稳定规则、工具说明、skill 说明和长期契约可能被动态材料、时间戳、路径、随机 id 或整段历史重排打断。备份只能保存现场，不能替代分析；如果备份前发现命中率明显偏低，必须留下原因、影响和是否需要回流到 feng 系统层 grow 的判断。

## Grow 单元，而不是 Session 产品心智

feng 不应该把“session”作为用户需要理解的核心概念。一个 grow 单元就是一个智能行为的成长边界：它包含目标、材料、运行契约、反馈、证据、下一轮 message list 和 hatch 候选。

在一个 grow 单元下可以有连续的内部运行过程，但产品心智上只有一个连续成长空间，而不是多个可切换 session。这样能保持 feng 的表面简单，也能避免用户把 feng 理解成普通聊天工具或 coding session 管理器。

这不否认内部需要记录轮次、事件、上下文版本或运行片段。区别在于：这些是 file native 成长状态的一部分，不是用户要管理的独立 session。

## 长程成长能力

feng 自身必须具备长程任务能力。这里的“长程”不是把上下文窗口变大，也不是把所有聊天历史压成一个摘要，而是让一个 grow 单元能在多次唤醒、中断、验证、反馈和材料补充之间保持判断连续。

对用户来说，这意味着：

```text
feng 知道当前目标是什么。
feng 知道还缺什么。
feng 知道上一次为什么失败或暂停。
feng 知道哪些反馈已经采纳，哪些只是候选。
feng 知道下一轮 LLM loop 将基于什么继续 grow。
feng 能在证据不足时等待、提问或阻塞，而不是假装完成。
```

所以 feng 的长程能力不是“无限自我运行”，而是可恢复、可审计、可验证的成长。每次 grow attempt 都应该能留下输入、message list、动作、证据和退出原因。这样用户才能相信它是在持续成长，而不是在一轮轮重新即兴发挥。

## 要解决的问题

今天创建一个专用 agent 往往会被迫进入一堆过早的工程概念：agent 框架、prompt 组织、工具协议、记忆系统、运行循环、调试接口、打包方式、版本升级。真正的问题是，创作者通常只是知道自己想要一个“会做某件事的智能行为”，却不想先成为 agent 工程师。

feng 要替代的是这条痛苦链路：

```text
想做一个智能行为
-> 手写 prompt、脚本、工具和胶水逻辑
-> 靠聊天记录或临时日志修正
-> 不知道它什么时候真的可用
-> 很难嵌入真实项目
-> 运行后暴露的问题无法回流
-> 新版本难以复制、升级和回滚
```

feng 的产品假设是：只要能把目标、世界入口、动作边界、运行结果、反馈归因和完成证据逐步沉淀下来，一个智能行为就能从模糊想法成长为可运行、可观察、可复制的产物。

## 目标用户

feng 至少有三类用户，产品心智必须分清。

创作者用户是想塑造智能行为的人。他可能是游戏设计师、小说作者、音乐创作者、开发者、自动化爱好者，或者任何想把某类智能行为嵌入项目的人。这个用户能提供目标、材料、样例、反馈和判断，但不应该被要求先理解 feng 的内部结构。

运行环境使用者是接入 hatch 产物的人。比如游戏引擎接入 boss agent，小说项目接入写作 agent，模拟器接入小车 agent。这个用户关心输入输出、稳定性、调试信息、版本和失败原因。

feng 自身维护者是用 feng 改进 feng 的人。feng 的递归演进必须通过真实下游 agent 暴露问题，再把系统性问题回流到 feng 自身，而不是靠抽象地声称“自我进化”。

## 核心体验

产品表面必须简单：

```text
说出你想要的智能行为。
给它材料、规则、样例和反馈。
feng 帮你试、问、改、验证。
成熟后，它变成能嵌入目标世界的运行单元。
运行中的问题会回流，继续推动成长。
```

feng 可以通过命令、对话式回合、IDE/终端环境或其他形式使用，但不应该被定义成一个沉重的聊天产品。对话只是澄清、反馈和修正的手段，不是所有 agent 的运行方式。

当用户输入模糊时，feng 不应该假装已经理解。它应该做三件事之一：

```text
基于明确假设先做一个候选版本。
列出阻塞当前成长的缺口。
只问最少的关键问题。
```

轻量不等于逃避模糊性。真正的轻量是：用户输入可以很轻，但 feng 对缺口、假设和证据的处理必须认真。

## Grow 与 Hatch 的交互心智

`grow` 和 `hatch` 不应该被理解成两个并列的用户动作。更合理的产品心智是：用户主要在持续推动 grow，hatch 是 grow 达到 readiness 后出现的成型结果。

理想体验接近：

```text
用户不断给目标、材料、样例、反馈和运行结果。
feng 每次 grow 推进一点：补齐门禁、整理材料、生成候选、运行验证、处理反馈。
当质量门禁、目标覆盖、运行契约和验证证据足够时，feng 说：成了，可以 hatch。
用户确认边界后，feng 才把稳定能力提取成 hatch 包。
```

因此，`hatch` 不应该变成一个让用户手动猜时机的命令。用户当然可以请求检查是否可 hatch，但系统不能因为用户输入了 hatch 就绕过 readiness。反过来，如果 readiness 已经满足，feng 应该主动收敛到 hatch 建议，而不是无限 grow。

这个交互心智能保留轻量感：用户不是在操作 agent 工厂，也不是在和一颗“蛋”对话，而是在持续塑造一个智能行为，直到 feng 给出可验证的“成了”。

## Grow

`grow` 是 feng 的成长过程。它不只是吸收聊天记录，也不只是改 prompt。

`grow` 至少同时塑造这些内容：

```text
行为能力：这个智能行为怎样完成目标。
运行契约：它接收什么输入，输出什么结果，运行在哪里。
世界感知：它能从哪些来源读取事实或状态。
动作边界：它能怎样影响外界，哪些动作需要限制。
可观察性：它运行后要记录什么，如何暴露问题。
反馈路由：哪些问题本地吸收，哪些上报上游，哪些忽略。
质量门禁：它怎样证明自己没有漏掉用户目标和目标世界要求。
文件状态：哪些成长材料、message list、证据和产物需要被写成文件。
```

因此，grow 的对象不是一个固定 LLM loop。目标 agent 的 loop、工具、技能、调试方式、运行接口、甚至是否使用 LLM，都可以随着 grow 被塑造出来。

但 grow 不能变成任意自我改写。基础审计、版本、来源、失败记录、权限边界、上报边界和关键文件记录必须稳定存在，否则系统会学会隐藏失败，而不是变得更好。

## 质量门禁与不能漏题

feng 必须把质量门禁当成 grow 的产物，而不是把一套固定评分表写死在系统里。

这点很关键。比如“写小说 agent 应该检查章节连续性、风格一致性、作者反馈处理”这类内容，不应该只是人类在文档里告诉 feng 的固定清单。真正的要求是：feng 在面对“我要做一个小说 agent”时，必须自己从用户目标、目标世界、材料、样例、反馈和 DoD 中产出当前项目的质量门禁，并在 grow 中持续维护。

质量门禁至少要回答：

```text
用户到底提出了哪些显式目标。
材料和目标世界隐含了哪些要求。
哪些要求是必须通过的硬门槛。
哪些要求只是优化方向。
哪些要求必须人工确认。
每个门禁项需要什么证据。
每个门禁项现在是未覆盖、待验证、已通过、失败、豁免还是需要人工判断。
```

这就是“不能漏题”。feng 不能只做自己顺手想到的部分，也不能因为某个候选结果看起来不错就宣布 ready。它必须维护目标覆盖表：所有进入 grow 的目标、材料、约束和反馈，都要被映射到门禁、运行契约、材料假设、反馈规则、待澄清问题、明确拒绝理由或上游/下游归属。

没有质量门禁，或者门禁没有覆盖关键目标时，系统不能进入 ready_to_hatch。此时正确行为是继续 grow、列出缺口、请求最少关键输入、等待证据或标记 blocked。

门禁可以有默认 skill 辅助，但默认 skill 只能提供维度和启发，不能替代当前 grow 单元自己生成门禁。小说、boss、小车、音乐 agent 的门禁都必须绑定当前目标世界，而不是套同一张通用评价表。

## Hatch

`hatch` 的产品含义不是“打包聊天历史”，也不是“一定导出一个 LLM agent”。它是把已经验证过的智能行为成型为目标环境可接入的运行包。

hatch 产物可能是：

```text
命令。
服务。
游戏组件。
脚本模块。
行为树。
状态机。
LLM + tools 的 agent。
创作工作流。
混合运行单元。
```

关键不在于形态统一，而在于产物必须带有明确运行契约：

```text
输入是什么。
输出是什么。
运行在哪里。
能感知什么。
能调用什么。
是否允许对话。
失败时怎么处理。
怎样被调试和观察。
怎样证明它可用。
怎样参与后续 grow。
```

对话可以是 hatch 产物的一种能力，但不能成为默认唯一接口。小说 agent 可能适合对话；游戏 boss agent 运行时更应该接收游戏状态并输出行为决策；智能小车 agent 应该接收传感器或模拟状态并输出控制动作。

## Hatch Agent 的质量底线

虽然 feng 不是 agent creator，但 hatch 出来的结果如果是 agent，就应该是一个优秀的 agent，而不是一段 prompt 加一层命令包装。

一个 agent 形态的 hatch 产物至少应该具备：

```text
清晰的运行任务和目标世界边界。
明确的输入、输出、动作和失败契约。
能从当前运行状态编译自己的 message list。
能区分当前观察、短期上下文、长期记忆和候选反馈。
能记录模型输入、输出、动作、工具结果和外部反馈。
能在调试模式下把问题形成反馈候选。
能在发布模式下保持版本锁定，不随意自我漂移。
```

这意味着 prompt 和上下文管理是 hatch agent 的内核能力之一。boss agent、小说 agent、小车 agent 可以有完全不同的运行方式，但都不能只靠一段不可检查的系统提示词维持能力。

## 世界感知与运行契约

“告诉 feng 如何感知世界”不是把任意材料塞进上下文。它应该转化为目标 agent 的运行契约。

用户可能提供：

```text
自然语言说明。
文件路径。
API 文档。
代码仓库。
日志。
网页。
截图。
游戏状态说明。
模拟器接口。
设备协议。
用户反馈。
运行报告。
```

这些材料只有被整理为以下内容，才算真正进入能力：

```text
可感知：从哪里读取事实或状态。
可行动：可以输出什么、调用什么、修改什么。
可约束：哪些动作需要限制、确认或降级。
可观察：运行时记录什么，如何暴露问题。
可验证：怎样证明结果可用。
可回流：问题应该进入哪一层 grow。
```

这里最大的概念风险是把“上下文很多”误认为“能力成熟”。真正的能力必须能运行、被观察、被验证，并能在问题出现时回到正确的成长层级。

## 可观察性与反馈路由

feng 的可复制性依赖一个关键能力：hatch 出来的东西必须能暴露自己哪里不够好，并把证据送回正确的成长层级。

运行问题不能无脑上报，也不能无脑吸收。每个 hatch 产物都应该具备反馈归因能力：

```text
哪些问题属于当前 agent 自己。
哪些问题属于创建它的 feng/grow 过程。
哪些问题属于 hatch 包装、版本、接口或调试能力。
哪些问题只是用户项目的上下文问题。
哪些问题是单次噪声，不应该吸收。
哪些反馈需要人工确认后才能进入新版本。
```

更稳的链路是：

```text
运行问题
-> 归因
-> 形成候选反馈
-> 附带证据
-> 本地或上游 grow
-> 验证
-> 版本化发布
```

这意味着 grow 不只塑造 agent 的行为，也塑造 agent 如何被观察、如何暴露问题、如何把运行经验变成下一轮成长材料。

## 默认反馈路由 Skill

多层回流闭环不应该只是 feng 的隐藏实现，而应该沉淀为 feng 默认携带的通用能力。概念上，它可以被称为“反馈路由 skill”。

这个 skill 的职责是判断运行问题如何在当前 agent、本地项目和上游 feng 之间归因与回流。它应该是默认存在的，但允许随着场景 grow。

它至少分成两层：

```text
基础协议：版本、来源、证据、上报路径、隐私边界、基础失败记录。
场景策略：具体收集哪些信号、如何判断归因、哪些问题上报、哪些本地吸收。
```

基础协议应该稳定，不能被下游随意破坏。场景策略可以成长：小说 agent、boss agent、小车 agent 需要不同的信号和归因方式。

feng 自己也应该通过真实多层场景 grow 这套 skill。比如：

```text
feng grow -> hatch 出 xiaoshuo
xiaoshuo grow -> hatch 出 libai 写作 agent
libai 写作 agent 在小说项目中运行并上报问题
xiaoshuo 吸收写作能力问题
feng 吸收反馈路由、hatch、调试和成长机制的问题
feng grow 改进默认反馈路由 skill
```

这不是抽象的“自我进化”，而是递归 dogfooding：feng 用自己产出的多层 agent 生态，反过来验证和改进自己的默认成长能力。

## 多层演进闭环

feng 的长期终态是多层闭环。

一个典型链路是：

```text
创作者提出智能行为
-> feng grow 出可运行候选
-> hatch 成运行包
-> 运行包进入真实项目
-> 调试/运行信息回流
-> feng 判断问题归因
-> 本地 agent grow 或上游 feng grow
-> 验证后产出新版本
```

以小说场景为例：

```text
F:\code\feng 中的 feng grow 自己
-> hatch 出新的 feng 或默认成长能力
-> 在 F:\code\xiaoshuo 中 grow 出小说 agent
-> hatch 出 xiaoshuo
-> xiaoshuo 在 F:\code\libai-chongshengle 中写作并开启调试
-> 写作问题先回流给 xiaoshuo
-> 系统性成长问题再回流给 feng
```

这种递归不是无监督无限演进。它必须依赖证据、归因、版本、验证和边界。上报不等于吸收，候选不等于合并，成长不等于发布。

### 小说场景的三层生命周期

在 `feng -> xiaoshuo -> libai-chongshengle` 这个案例里，三个目录不是简单复制关系，而是三个不同生命周期层级：

```text
F:\code\feng
  feng 自身的成长项目。
  负责 grow/hatch feng 的默认能力、Grow Kernel、Agent Runtime Kernel、feedback router skill 和 hatch contract。
  只吸收系统性问题，不直接吸收某一本小说的内容问题。

F:\code\xiaoshuo
  小说 agent 的成长项目。
  负责 grow/hatch 一个可复制的小说创作 agent。
  吸收反复出现的小说创作能力问题，比如长篇上下文、章节计划、风格一致性、作者反馈转修订。

F:\code\libai-chongshengle
  具体作品项目。
  负责维护《李白重生了》的世界观、人物、提纲、章节、修订和作者反馈。
  使用 xiaoshuo 运行包写作，并把运行问题形成反馈候选。
```

这只是概念结构示意，不是最终目录 schema。它要表达的是：不同生命周期的数据不能混在一起。作品事实默认留在作品项目，领域 agent 问题回流给 xiaoshuo，系统性 grow/hatch/feedback/kernel 问题才提议给 feng。

### 小说场景的终态流程

终态流程可以理解为：

```text
1. feng 在 F:\code\feng 中 grow 自己，形成默认 kernel、skill、hatch 和反馈路由能力。
2. feng 在 F:\code\xiaoshuo 中 grow 小说 agent，定义小说创作的输入、输出、上下文、动作边界、质量门禁、验证和反馈路由。
3. xiaoshuo hatch 成小说 agent 运行包。
4. xiaoshuo 被放入 F:\code\libai-chongshengle，读取作品设定、提纲、已有章节和作者目标，并为当前作品产出作品层质量门禁。
5. xiaoshuo 生成章节或修订，并记录本轮 message list、输出、trace、门禁结果、验证结果和反馈候选。
6. 作品层门禁失败先留在 libai-chongshengle；反复出现的写作能力门禁失败回流给 xiaoshuo；系统性机制门禁失败才上报给 feng。
7. xiaoshuo 或 feng 经过 grow、验证和 hatch 后，发布新版本能力包。
```

数据流应该是双向但分层的：

```text
feng -> xiaoshuo：默认 grow 能力、feedback skill、agent runtime kernel、hatch contract。
xiaoshuo -> libai-chongshengle：小说 agent runtime、写作 workflow、作品接入契约、调试上报能力。
libai-chongshengle -> xiaoshuo：章节失败、风格偏差、设定冲突、作者反馈、运行 trace 摘要。
xiaoshuo -> feng：上下文管理、反馈路由、hatch 包、runtime kernel 等系统性问题。
```

明确不应该发生的是：`libai-chongshengle` 的全部作品内容默认进入 `feng`，某个作品的设定变成 `xiaoshuo` 的通用规则，或者某次作品失败直接改写 feng 的默认 kernel。

更完整的小说场景案例见 `docs/feng-novel-case-flow.md`。

## 示例：Boss Agent

如果创作者想做一个 boss agent，feng 的理想成长过程不是先选择模板，而是先塑造玩家体验，再压成游戏运行时可用的决策能力。

成长过程可能是：

```text
创作者提出 boss 想法。
feng 抓住 boss 的体验目标、弱点、阶段和不公平边界。
feng 绑定游戏世界：玩家距离、血量、技能冷却、场地、阶段等。
feng 生成可运行候选行为。
候选 boss 在模拟/测试场景中运行。
boss-agent 调试模式上报状态、动作、原因、结果和异常。
创作者反馈：太黏人、二阶段无聊、被远程卡死、低血量行为不对。
feng 把反馈归因为本地行为问题或上游成长机制问题。
多轮 grow 后，hatch 出可嵌入游戏的 boss-agent 包。
```

boss-agent 包不一定是一个实时 LLM loop。它可能是行为树、状态机、策略脚本、受限 LLM 决策器或混合模块。游戏运行时需要稳定、低延迟、可调试、可复现，因此 hatch 形态必须服务于目标世界，而不是服务于某个固定 agent 架构。

## 安全边界

feng 可以有行动力，但不能默认无限行动。越强调成长、回流和递归演进，越需要真实边界。

必须成立：

```text
读写文件有工作区边界。
运行命令有权限边界。
访问网络、外部服务或设备有授权边界。
删除、发布、写外部系统、控制硬件等高风险动作需要确认。
密钥和本机私有配置不能进入发布包。
从外部仓库带来的 feng 实例不能默认可信。
运行记录和上报内容需要隐私边界。
自动更新必须可确认、可回滚、可复现。
```

file native 不等于无限暴露。文件里的运行记录、message list、调试上报和成长材料可能包含私有信息，必须有边界、脱敏和发布排除机制。

开发期可以允许更快的 grow 和候选迭代。发布期必须锁定版本，升级需要明确证据和确认。否则 agent 的行为会不可复现，也无法被信任。

## 当前可行性判断

这个产品概念有价值，但必须避免过度承诺。

可行的方向是：

```text
feng 作为受约束的 agent 成型与回流系统。
从一个窄场景开始，把智能行为 grow 成可运行包。
hatch 产物带运行契约、可观察性和反馈路由。
真实运行问题通过归因回流到本地或上游。
feng 通过多层 dogfooding 改进自己的默认成长能力。
```

不可直接承诺的是：

```text
任意自然语言输入都能自动变成可靠 agent。
任意世界都能被无损感知。
任意 hatch 产物都能无人工确认自动升级。
目标 agent 可以随意改写自己的审计和上报底座。
feng 可以无监督地无限自我演进。
```

第一阶段应该选择一个失败成本低、反馈丰富、能暴露成长机制问题的场景。小说 agent 很适合验证多层回流，因为它会暴露世界观记忆、风格一致性、反馈归因、hatch、调试和版本更新等核心问题。游戏 boss agent 很有价值，但更依赖模拟测试、运行接口和可复现性，不适合作为毫无约束的第一个证明。

## 需要避免的思维误区

聊天历史不是能力。只有能运行、能约束、能观察、能验证、能回流的内容，才算进入能力。

可复制不是复制 prompt 或成长历史。可复制是复制一个带运行契约、可观察性、反馈路由和版本边界的运行单元。

对话不是所有 agent 的运行方式。对话适合创作、澄清和反馈，但 boss、小车、服务型 agent 需要面向目标环境的接口。

hatch 不是导出一个固定 LLM loop。hatch 导出的是目标世界可接入的运行包，形态可以随 grow 成型。

反馈回流不是无脑吸收。上报要有归因，吸收要有证据，合并要有验证，发布要能回滚。

默认 skill 不是不可变规则。基础协议要稳定，场景策略可以 grow。

质量门禁不是固定评分表。门禁必须由 grow 过程产出，绑定当前目标和材料；默认门禁 skill 只能辅助，不能替代当前项目的门禁生成。

不能漏题不是提示词，而是 readiness 前必须检查的覆盖关系。未覆盖的目标、材料约束或用户反馈，必须阻塞 hatch 或发布。

file native 不是把一切垃圾都写进仓库。它要求关键成长状态可追踪，但仍然需要筛选、脱敏、压缩和清理。

自我演进不等于无限放权。feng 的递归演进必须通过 dogfooding、证据、版本和边界发生。

## 非目标

以下内容暂不在概念阶段决定：

```text
具体目录结构。
具体文件 schema。
metadata 格式。
candidate 和 validated 的精确机制。
工具协议细节。
MCP adapter 细节。
eval runner 细节。
message compiler 细节。
provider adapter 细节。
hatch package 目录布局。
file native 的具体目录布局。
message list 的具体文件格式。
CLI 子命令完整设计。
GUI 设计。
插件市场或分发平台。
游戏引擎接入协议。
自动更新实现机制。
```

这些都应该在产品概念稳定后再进入设计。现在更重要的是确认 feng 到底承诺什么、不承诺什么，以及用户如何判断它真的有用。

## 当前开放问题

1. 第一个可公开证明的场景应该选小说 agent，还是游戏 boss agent 的开发期 grow/hatch 流程？
2. 默认反馈路由 skill 的基础协议应该包含哪些最小信息，才能既可信又不变重？
3. 哪些运行信息可以被场景策略 grow，哪些审计信息必须稳定不可绕过？
4. hatch 产物的“可运行包”在第一阶段应该以什么形态出现：命令、脚本模块、服务，还是宿主环境内的 skill？
5. feng 自我迭代时，哪些下游问题允许上报到 feng-core，哪些必须留在目标 agent 本地？
6. 自动更新在开发期和发布期应该有什么不同边界？
7. 下一轮 LLM loop 的 message list 应该如何由文件化成长状态编译出来，并让用户能检查其依据？
8. 质量门禁和目标覆盖表如何文件化，才能防止漏题但不把用户拖进繁重配置？

这些问题不需要在本文一次性解决，但它们决定 feng 是否能从一个有吸引力的概念，变成一个用户敢持续使用、敢嵌入项目、敢递归演进的系统。
